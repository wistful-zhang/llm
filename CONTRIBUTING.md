# 参与贡献

感谢帮助完善大模型面经。这个仓库同时包含网站模板和示例题库，请根据目标选择合适的方式。

## 建立自己的题库

请使用仓库页面上的 **Use this template**，不要通过 Pull Request 把个人笔记提交到公开题库。模板创建的仓库完全独立，可以选择公开或私有。

## 向公开题库贡献

可以通过两种方式：

1. 从网站“面经社区”使用 GitHub Issue 表单提交题目和参考答案，其他人可以直接在帖子下评论补充，维护者整理后收录。
2. Fork 仓库、增加或修改 `docs/_questions/` 中的内容，然后提交 Pull Request。

普通贡献者不需要，也不应获得 Pages CMS 或仓库写权限。Pages CMS 直写只适合题库主人和明确受信任的长期编辑；临时投稿统一走 Issue 或 Fork + Pull Request，便于公开检查和撤回权限。

社区帖子和正式题库是两个状态：投稿提交后会立即公开并允许评论，不需要等待才能被看到；只有收录进 `docs/_questions/` 前，维护者才会核对隐私、授权、来源和技术事实。公开仓库中的单条 Issue 不能设置为私有，包含敏感信息的面经请留在本地或自己的 Private 仓库。

## 内容标准

一份高质量题目建议包含：

- 针对这道题的一两段答题提示：怎么开口、讲到什么程度、面试官可能从哪里追问
- 一段自然、可直接说出口的参考回答；直接回答问题，不写“我先讲结论”之类的过程旁白
- 准确但不要求逐字背诵的核心回答
- 原理、流程或公式的展开说明
- 工程实践中的选择、指标与取舍
- 至少 3 个常见追问，并在每个问题后给出 1～2 句直接接法
- 一句话复习结论

不同题型不必套同一结构：原理题讲清因果链，系统题先交代约束和失败路径，项目题只使用真实经历与数据。简单问题答到位即可，不需要为了凑满 60 秒补一段项目落地。

请优先使用自己的表达并核对事实。不要提交：

- 公司内部资料、未公开题库或受保密协议约束的内容
- 个人信息、账号凭据、访问 Token 或其他秘密
- 未获授权的大段版权内容
- 与大模型面试无关的推广信息

整理公开面经时，请只提炼高频主题并重新组织答案，在正文末尾列出可访问的公开来源与主要技术依据。匿名社区帖子适合说明“有人被问到过”，不能单独证明它是某公司的固定真题；不要把汇总题库标成未经核实的公司原题。

维护者完成题目来源、技术事实、七段答题结构、追问接法和参考链接检查后，可以加上 `verified: true`。它会在阅读网站显示“资料核验”标记，并触发更严格的自动校验；普通速记和个人题目保持省略或 `false` 即可。

## 文件格式

复制 `docs/_templates/question.md`，文件名使用简短、稳定的英文或数字 slug。推荐优先使用以下常见分类，也可以增加含义明确的新分类：

- LLM 基础
- 训练与对齐
- RAG
- Agent
- 工程实践
- 系统设计

提交带完整答案的公开题目应使用正式分类，难度必须是 `简单`、`中等` 或 `困难`，并设置 `review_status: 待复习`（或 `已掌握`）、`published: true` 与 `answer_status: complete`。仅提交问题时可以使用 `answer_status: pending`；正文即使有半成品也不会在阅读网站展示或进入模拟面试。

## 提交前检查

- 页面标题、分类、难度、复习状态、标签和日期完整
- 内部链接有效，代码块指定合适语言
- 没有敏感信息和无权公开的内容
- 一个 Pull Request 尽量只处理一个主题

## 贡献授权

提交 Issue 或 Pull Request 前，请确认内容由你原创、经过允许，或来自允许再分发的资料。提交即表示你有权公开该内容，并同意本项目按照根目录的 MIT License 保存、修改和分发这次贡献。引用论文、官方文档或公开面经线索时，请保留来源链接，不要复制大段受版权保护的原文。
