## 面试时怎么答

建议按“结论 → 原理 → 取舍 → 落地”回答：

1. **先给结论**：先把 RAG 拆成离线建库和在线问答两条链，不要只回答“向量库加 LLM”。
2. **再讲关键机制**：沿解析、切块、索引、查询处理、召回、重排、组装、生成和引用顺序展开。
3. **主动说取舍**：指出召回率、上下文噪声、延迟和可追溯性相互制约，RAG 仍可能幻觉。
4. **最后落到项目**：以 Recall@K、Faithfulness、P95 延迟、引用命中率分层验收；说完停顿让对方选一层深挖。

**60 秒口述示例：**

> 我会先说 RAG 有两条链：离线把文档解析、切块、向量化并建立带权限和版本的索引；在线先处理查询，再召回、融合重排、按 Token 预算组装证据，最后让模型生成并附引用。它的优势是知识可更新、可追溯，但检索正确也不保证生成忠实。项目中我会分层看 Recall@K、重排 nDCG、答案 Faithfulness、引用命中率和 P95 延迟，错误时先判断是没找着还是没用好证据。

## 核心回答

RAG 通常分为离线建库和在线问答两条链路。离线阶段完成数据采集、解析、切块、向量化并建立索引；在线阶段处理用户问题，在当前 tenant、用户和 ACL 约束下召回候选片段，再去重、融合或重排后组装上下文，让模型基于问题和证据生成回答。RAG 给模型提供了可更新、可追溯的外部信息，但不能保证模型一定忠实使用这些信息。

## 展开说明

一条常见链路可以拆成：

1. **数据准备**：解析网页、PDF、数据库等来源，保留标题、页码、版本和权限元数据。
2. **索引构建**：按语义边界切块，用 Embedding 模型编码，并写入向量或关键词索引。
3. **查询处理**：规范化问题，必要时补全指代、改写或拆分查询。
4. **检索与排序**：获取较大的候选集，再过滤、融合和重排，选出最终证据。
5. **上下文组装**：控制 token 预算，保留来源标识，并减少重复或冲突片段。
6. **生成与校验**：要求模型依据证据回答；对答案、引用和拒答条件做检查。

原始 RAG 将模型参数视为参数化记忆，将外部索引视为非参数化记忆。生产系统中的检索器、索引和生成器可以独立升级，因此排错时也应分层评估。

## 工程实践

最小可用版本先做到“可回放”：记录数据版本、原始查询、候选文档、最终上下文、模型版本和答案。回答错误时先判断证据是否被召回，再判断生成器是否正确利用证据，避免把所有问题都归因于 Prompt。

## 常见追问

1. **RAG 为什么能更新知识而不重新训练模型？** 知识保存在外部索引中，更新文档与索引即可让下一次推理读取新证据，模型参数不必随每次知识变化而更新。
2. **离线建库和在线查询分别有哪些性能瓶颈？** 离线常卡在解析、Embedding 吞吐和索引构建；在线常卡在查询改写、向量检索、重排和 LLM Prefill，并要关注尾延迟。
3. **已经召回正确文档，为什么仍可能回答错误？** 关键片段可能在切块或打包中丢失，也可能被噪声淹没；生成器还可能忽略、误读或与参数记忆冲突。

## 一句话复习

> RAG 是“建库、检索、组装证据、生成与校验”的完整链路，不只是向量数据库加一次模型调用。
