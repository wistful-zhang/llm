## 面试时怎么答

建议按“结论 → 原理 → 取舍 → 落地”回答：

1. **先给结论**：先说 MoE 用 Router 为每个 Token 选择少量专家，以稀疏计算扩大参数容量；停顿后讲负载。
2. **再讲关键机制**：解释 Top-k、容量因子、Token Dispatch、辅助均衡损失与丢弃/溢出处理。
3. **主动说取舍**：指出均衡太弱会热点，太强会压制专家化；Top-2 更稳却增加计算与通信。
4. **最后落到项目**：按专家和设备观测负载，监控路由熵、溢出率、最大/平均负载、通信占比和任务质量。

> 我的结论是，MoE 通过 Router 为每个 Token 只激活少数专家，让参数量变大但单 Token 计算保持可控；真正难点是负载不均和跨设备通信。这里停一下，再讲 Top-k 路由、容量因子和辅助均衡损失：容量太小会丢 Token，均衡损失太强又会让专家失去专门化。Top-2 质量与容错更好，但计算和 All-to-All 更贵。项目里我会逐层看路由熵、专家最大/平均负载、溢出率、通信时间占比、每秒 Token 和下游质量。

## 核心回答

稀疏 MoE 通常把 Transformer 中的部分 FFN 替换成多个专家。Router 根据每个 token 的表示计算专家分数，只激活 Top-1 或 Top-k 专家，再按路由权重合并输出。这样总参数量可以大幅增加，而每个 token 只经过少数专家，计算量不会随专家总数线性增长。主要难点是热门专家过载、冷门专家学不到数据，以及跨设备 All-to-All 通信；常用负载均衡辅助损失、容量限制和路由正则来缓解。

## 展开说明

Router 本质上是一个可学习分类器，但其目标不只是选“最合适”专家，还要让系统可训练、可执行。若大量 token 都去同一个专家，该专家超过容量后可能丢弃、绕过或延迟处理 token，同时其他专家闲置。辅助损失通常鼓励路由概率和实际 token 分配更均匀，但权重过大又可能牺牲专业化。

MoE 的“稀疏”主要发生在专家 FFN，Attention 等模块仍可能是稠密计算。总参数增大也会增加权重存储、检查点和通信成本，因此“相同 FLOPs 下更多参数”不等于部署成本不变。

## 工程实践

训练时应监控每个专家的 token 数、容量溢出率、路由熵、负载变异系数和通信时间，并按数据领域切片检查是否出现专家塌缩。专家并行的布局要结合网络拓扑设计；如果 All-to-All 成为瓶颈，理论上的稀疏计算优势可能无法转化成端到端吞吐。

## 常见追问

1. **Top-1 与 Top-2 路由在质量、计算和容错上有什么差异？** Top-1 只激活一个专家，成本低但对路由错误更敏感；Top-2 融合两个专家通常更稳，却增加计算和通信。
2. **负载均衡损失过强会带来什么问题？** Router 会为了均匀分配牺牲语义匹配，抑制专家专门化，甚至损害主任务质量。
3. **为什么 MoE 参数量很大，单 Token FLOPs 却可以接近较小的稠密模型？** 每个 Token 只经过少数激活专家，大量非激活参数不参与该 Token 前向；但通信与路由开销仍存在。

## 一句话复习

> MoE 用 Router 为每个 token 稀疏选择专家，以更多总参数换取有限激活计算，但必须治理负载和通信。
